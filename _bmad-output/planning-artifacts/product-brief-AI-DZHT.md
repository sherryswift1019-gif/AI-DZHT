# Product Brief: AI-DZHT — AI 研发指挥台

## Executive Summary

软件工程正在经历一场安静的危机：每个工程师都在用 AI 写代码，却没有人知道 AI 在整个团队里究竟做了什么。需求是否被正确理解？架构决策是否经过评审？代码是否在任务范围内？交付质量有没有系统性保障？这些问题的答案，藏在每个工程师自己的对话框里，对团队完全不透明。

AI-DZHT 是面向研发团队的 AI 研发流水线平台。它用一个指挥官 Agent 统一调度研发全生命周期 —— 从需求理解、架构设计、代码实现到测试与部署 —— 每个阶段由专职 Agent 独立执行，人类只在关键决策节点介入审批。更重要的是，整个过程完全可观测：每个 Agent 在做什么、消耗多少 token、产出什么质量，实时可见。

这不是又一个"AI 帮你写代码更快"的工具。这是让研发过程第一次真正透明化、可管理、可衡量的指挥平台。

---

## 问题

在拥有 80 名工程师的团队里，AI 辅助研发已经是日常。但这带来了一个新的管理困境。

**黑盒研发**。没有人知道需求是否被 AI 正确理解过，架构决策是否经过任何形式的评审，开发 Agent 是否在任务范围内工作。每次出问题，只能事后追溯，无法事前预防。

**投入产出失衡**。团队花了大量时间和 token 成本在 AI 工具上，但无法量化收益。哪些需求效率最高？哪个阶段是瓶颈？哪类任务最容易跑偏？这些问题没有答案。

**规范无法落地**。代码规范、架构原则、业务约束 —— 这些标准每个团队都有，但 AI 时代每人开一个对话框，规范根本无法系统性地注入到研发流程中。靠人工复查、靠经验判断，在 80 人团队是一条逐渐失控的路。

---

## 解决方案

AI-DZHT 以 Claude API + prompt 工程为技术底座，在现有 GitHub/GitLab + Jenkins + K8s 工具链之上叠加能力，无需重建基础设施。

**流水线结构（7 个 Agent）**：

| Agent | 职责 |
|---|---|
| 需求 Agent | 接收需求文档，生成用户场景和任务拆分清单 |
| 设计 Agent | 产出功能规格和交互说明 |
| 架构 Agent | 输出技术方案和系统设计 |
| 开发 Agent | 按任务拆分编写代码，commit 级任务合规检查 |
| 测试 Agent | 生成测试用例，执行测试，输出质量报告 |
| 部署 Agent | 触发 Jenkins 流水线，通过 K8s 发布 |
| **指挥官 Agent** | 全局调度、状态追踪、阶段放行、异常升级 |

**人机协作**：系统默认全自动推进，人类只在预定义关键节点（如需求确认、架构评审、发布上线）审批。指挥官是唯一交互界面 —— 人只需要看报告、拍决策。

**可观测核心**：每个 Agent 的执行状态、输入输出、token 消耗、产出质量结构化上报，研发大盘一屏可见。

---

## 差异化

**卖可控感，不卖速度**。市场上 AI 研发工具的主流叙事是"更快写代码"。AI-DZHT 的主张是"第一次看清楚研发在发生什么" —— 对质量焦虑的团队和需要向上汇报 ROI 的管理层，这是截然不同的价值主张。

**规范内嵌于流程**。代码规范、架构原则、业务约束通过 Agent prompt 和知识库系统性注入，不再依赖每个工程师记住并执行。

**BMAD 方法论作为能力底座**。每个专职 Agent 直接继承 BMAD 的专业能力，冷启动即"足够专业"，团队只需要教会系统自己的业务规则。

**知识积累护城河**。每次人工推翻 Agent 决策、每次生产事故，都触发经验提取，写入知识库。系统越用越懂业务 —— 这是时间带来的竞争壁垒，无法被快速复制。

---

## 目标用户

**研发团队负责人（主要用户）**
需要 AI 研发投入有清晰 ROI 感知，在不 micromanage 的情况下保障交付质量。
成功信号：能在一张大盘看清所有需求状态，能量化每分钱的 AI 投入产出。

**研发工程师（日常用户）**
不再需要重复解释上下文，不再需要记住所有规范。
成功信号：跑通一个需求只需处理少数几个关键审批，其余全部自动。

**产品经理（轻度用户）**
提交需求，跟踪进展，不再频繁拉会对齐。
成功信号：需求进去，看着它在大盘里一步步走向上线。

## 成功标准

**MVP（第 1 个月）**
- 完整流水线端到端跑通（需求 → 设计 → 架构 → 研发 → 测试 → 部署）
- 指挥官 Agent 能调度各阶段，关键节点人工审批可配置
- 基础可观测：Agent 状态、token 消耗、输出在大盘可追踪
- 至少 1 个真实需求从头跑到尾，产出可部署代码

**上线后 3 个月**
- 团队能量化每个需求的端到端交付时间
- Token 成本可追踪，AI 投入 ROI 可估算
- 需求返工率相较基线有可测量的下降
- 关键审批节点的平均响应时间 < 4 小时（不成为流程瓶颈）

---

## MVP 边界

**包含**：
- 7 个 Agent（6 个专职 + 1 个指挥官）
- 研发大盘（Agent 状态、token、产出追踪）
- 人工审批节点（可配置）
- GitHub/GitLab 集成（commit 触发、PR 联动）
- Jenkins + K8s 部署集成
- 钉钉通知（审批提醒、状态更新）

**明确不在 MVP 内**：
- 知识积累与经验沉淀系统
- "圆桌作战室"可视化 UI（MVP 用简单大盘）
- 提示词风险过滤层
- 需求变更外科手术引擎
- 多项目并行管理

---

## 未来愿景

三年后，AI-DZHT 成为研发团队的"数字神经系统" —— 每一个需求的诞生、演进、上线、退役，都有完整的 AI 执行记录和知识沉淀。团队中每个人都是产品负责人，核心竞争力不再是写代码的速度，而是清晰表达需求、判断 Agent 输出质量、在关键时刻做出正确决策的**元判断力**。

长远来看，这套方法论和工具链有可能演化为面向行业的研发效能平台 —— 将 AI-DZHT 的内部实践，变成可复制的研发操作系统。
