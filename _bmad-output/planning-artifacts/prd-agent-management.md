## 执行摘要

AI-DZHT Agent 管理模块是面向研发团队的 **Agent 模板注册表与能力编排系统**，是 AI-DZHT 平台的核心基础设施模块，也是项目管理、需求执行流程与指挥官 Agent（Commander）的上游依赖——其他模块的正常运行以本模块为前提。

本模块定位于**能力定义层**：管理 BMAD 命令库、Agent 模板配置、提示词、能力编排与上下文注入规则；不承担 Agent 运行时执行质量保障（由各 Agent 自身及工作流执行模块负责）。

**核心架构决策：模板 / 实例分离**

Agent 模板为配置层，定义能力与行为规范；每个需求执行时生成独立隔离的 Agent 实例（运行层），实例间完全隔离、互不干扰。模板修改不影响运行中的实例，历史执行记录完整可追溯。**Commander** 在每次需求执行时基于本模块的 Agent 模板库动态调度调用顺序——本模块向 Commander 提供可用 Agent 列表；Commander 基于列表发起实例化请求，由工作流执行模块执行，本模块接收执行模块推送的实例状态变更通知。

**目标用户及主要接触场景：**

- **平台管理员**：维护全平台 Agent 模板库与 BMAD 命令库、管理共享权限与版本、审批共享 Agent
- **项目/团队负责人**：为项目创建定制 Agent 变体、配置项目级上下文，决定 Agent 组合
- **研发成员**：主要接触点为"查看可用 Agent 目录"和"了解当前项目配置"，不主动管理模板；受益于平台预置 8 大研发角色 Agent，零培训直接使用

**核心问题陈述：**

① 能力散落，经验无法沉淀复用；② Agent 能力边界不清晰，团队缺乏一致预期；③ 没有治理机制，优质配置无法团队流通。Agent 管理模块通过结构化模板注册表系统性解决上述问题。

### 差异化核心

AI-DZHT Agent 管理模块与通用 Agent 平台（Dify、Coze、AutoGen Studio）的本质区别：

1. **角色预置消化认知负担**：平台内置 8 大研发角色 Agent，用户无需定义"Agent 是什么"，直接进入"如何为项目配置使用"——零培训来自认知负担已被平台提前消化。

2. **BMAD 命令库作为原子能力单元**：命令库（BP/DR/CA/DS/QA 等）是所有 Agent 能力的最小构成单元，独立管理、统一维护，Agent 能力编排以命令库为基础——确保能力定义标准化、可扩展。

3. **三层上下文注入架构**：Agent 基础提示词（模板层）+ 项目上下文（行业/技术栈）+ 需求上下文（PRD 摘要/Story 列表），各层独立配置、按需叠加，Agent 行为随上下文自适应。

4. **团队级可治理资产**：Agent 模板支持三级共享范围（团队私有 / 平台开放 / 申请使用），版本快照、历史回滚、变更锁定全覆盖——AI 能力可治理、可复用、可追溯。

### 交付分期

本模块分三期迭代交付：

- **Phase 1 / MVP（P0）**：核心 Agent 模板管理、BMAD 命令库、三步创建向导、AI 生成提示词、生命周期管理、三层上下文配置
- **Phase 2 / 完整 P0**：命令库扩展、Agent 详情实例视图、相似度对比卡、会话回放、共享申请流程
- **Phase 3 / P1 成长**：跨团队模板市场、Agent 效能分析、高级版本 Diff

> **术语说明：** 本文档中 "Phase 1 / MVP / P0（MVP）"、"Phase 2 / 完整 P0" 等短语互为同义。下文统一使用 Phase 1 / 2 / 3 / 4 标记分期，P0/P1/P2 标记功能优先级。

## 产品范围

### MVP — 最小可用产品（核心闭环）

**最小可用闭环：能力定义 × 零门槛创建 × 上下文智能 × 系统保护**

| 功能 | 说明 | 纳入原因 |
|------|------|----------|
| F01 命令库基础管理 | 查看内置命令，内置命令不可删 | 能力编排的原子单元 |
| F03 能力按研发阶段分组 | 分析/规划/架构/实现四阶段分组展示 | 零培训上手的认知基础 |
| F04 三步创建向导 | 基本信息 → 能力勾选 → 提示词确认 | 主要创建路径 |
| F05 AI 生成提示词草稿 | 根据所选能力自动生成结构化草稿 | < 10 分钟配置的核心依赖 |
| F22 项目上下文配置 | 行业/技术栈/团队规范，一次配置全项目复用 | 三层上下文注入的必要层 |
| F10 内置 Agent 保护 | 8 个系统 Agent 标记保护，不可删除或覆写 | 数据完整性安全底线 |
| 基础状态展示 | Agent 列表显示运行中/空闲/已停用状态 | 用户必须知道"能否操作"，防止误操作 |

**MVP 验证假设：** 用户能在 10 分钟内完成"选能力 → 生成提示词 → 绑定项目上下文"闭环，且系统 Agent 不会被误操作破坏，Agent 管理模块的核心价值即已验证。

### 旅程 1 — 项目负责人：第一次从零配好一套 Agent（核心价值旅程）

**角色：李明，电商平台研发负责人**
刚接手"会员体系重构"项目。每次用 AI 都要重新解释业务背景，写了十几分钟的说明，AI 回的东西还是通用的。他今天第一次打开 Agent 管理模块。

**场景展开：**

李明打开 Agent 管理页，8 张 Agent 卡片清晰陈列，每张有能力说明和状态徽章"空闲"。他点击"为项目配置上下文"——行业领域（电商）、核心技术栈（React + Java + MySQL）、团队规范（API 先行，禁止直接操作数据库）。保存后系统提示："此后本项目所有 Agent 实例将自动携带此上下文。"

他创建"电商产品 Agent 变体"，三步向导，步骤 2 勾选 BP、DR、CP 命令；步骤 3 AI 自动生成提示词草稿。

**分支 A — AI 草稿质量满意**：草稿头两句准确，他微调两句，10 分钟完成配置。

**分支 B — AI 草稿质量不佳**：草稿太通用，没有体现电商特性。提示词编辑区显示四块结构化编辑：① 身份定义 ② 能力边界 ③ 行为规范 ④ 上下文规则。李明定位到"① 身份定义"块，点击"单块 AI 优化"，输入补充说明"聚焦电商转化漏斗分析"，AI 重新生成该块。其余三块保持不变，15 分钟完成配置。

**啊哈时刻**：无论哪条分支，李明都完成了。他打开第一个需求，Agent 直接进入状态——"这个 AI 知道我们是做电商的。"

**旅程揭示的需求：**
- F22 项目上下文配置、F04 三步向导、F05 AI 提示词生成
- 四块结构化编辑区（分块独立维护 + 单块 AI 优化）
- 降级路径：AI 草稿不满意时的手动补救入口

### 旅程 2 — 研发成员：执行前了解配置 + 发现异常后反馈（消费者旅程）

**角色：小王，前端工程师**
被分配到"会员等级展示"需求，想在开始前知道搭档是谁、边界在哪，执行中发现 Agent 行为与预期不符。

**场景展开：**

小王进入需求详情，"本需求使用的 Agent"面板列出 3 个 Agent 及其能力边界——他知道了接口联调不是自己等 AI 做，放心开始工作。

**执行中异常**：前端 Agent 生成的组件没有遵守团队"禁止内联样式"规范。小王发现了，但他没有编辑权限。他点击 Agent 卡片上的"反馈问题"按钮，填写："行为规范 ③ 未覆盖内联样式限制"，提交给项目负责人李明。

李明收到通知，查看详情，确认后进入编辑模式补充规范。下次执行自动生效。

**闭环时刻**：小王的发现没有被忽略，也没有绕过系统自行解决——Agent 配置因此更准确，下一个需求直接受益。

**旅程揭示的需求：**
- 需求级 Agent 视图（能力摘要 + 边界说明）
- 研发成员只读权限
- Agent 卡片"反馈问题"入口 → 通知项目负责人

### 旅程 3 — 平台管理员：发现冗余 Agent，治理清理（管理治理旅程）

**角色：张超，平台管理员**
平台运行 3 个月，Agent 从 8 个增长到 31 个，其中多个名称相似、能力高度重叠。

**场景展开：**

张超打开能力反查（F12），输入"CP"，列出 9 个含此能力的 Agent，系统自动标记 3 组"高重叠对"，并显示各自 30 天使用情况。他停用零调用的 Agent——系统扫描依赖后提示"无关联运行中工作流，可安全停用"。确认操作，历史记录保留。

**成就感时刻**：Agent 数量从 31 降到 22，冗余率从 45% 降到 12%。平台 AI 能力第一次"有人管了"。

**旅程揭示的需求：**
- F12 能力反查、F13 重叠标记、F17 停用依赖扫描
- 使用情况统计（调用次数/最近活跃）

### 旅程 4 — 跨团队共享：发布与申请使用（P1 共享旅程）

**角色 A：陈静**，金融团队负责人，打磨出"合规优先型产品 Agent"
**角色 B：赵峰**，另一个金融项目负责人，想使用陈静的 Agent

**场景展开：**

陈静：共享设置选"需申请使用"，勾选"禁止二次共享"，填写使用说明，发布。Agent 卡片出现"共享-申请制"徽章。

赵峰：看到目录里的 Agent，申请使用，填写理由，提交。陈静审批通过，赵峰项目出现该 Agent，标注来源"金融团队 · 陈静"。

**Fork 版本独立性**：赵峰可以 Fork 出变体供本项目调整。Fork 后的变体**完全独立**——陈静后续更新原版，赵峰的变体**不会自动同步**。变体详情页标注"基于原版 v2，当前原版为 v4，可手动查看差异并决定是否合并"。赵峰主动查看差异后，选择手动同步其中一处改动。

**知识流通时刻**：陈静两周的打磨结果在平台流通；赵峰不用从零开始；Fork 独立性保证定制不会被意外覆盖。

**旅程揭示的需求（P1）：**
- F06 三级共享范围、F23 申请审批流、来源溯源展示
- Fork 后版本完全独立；原版更新不自动同步
- 变体详情页显示"与原版版本差异"及手动合并入口

### 旅程 5 — 边缘场景：尝试修改运行中的 Agent（异常保护旅程）

**角色：刘芳，项目负责人**
发现"架构 Agent"提示词描述不准确，想立刻修改，但不知道有 3 个需求正在用这个 Agent。

**场景展开：**

刘芳点开编辑区——按钮灰色，提示"当前有 3 个运行中实例，提示词不可修改"，并列出 3 个需求实例及其状态。她停止已暂停的需求 C 实例，等 A、B 完成后，锁定自动解除。她完成修改，系统自动保存新版本，旧版快照保留。

**安全感时刻**：系统保护了她，而不是限制了她——没有在不知情的情况下破坏运行中的工作流。

**旅程揭示的需求：**
- F18 运行中锁定 + 锁定状态徽章
- 运行实例列表与影响说明
- 手动停止实例入口、F21 自动版本快照

### 旅程 6 — 新成员接手已有项目：理解既有 Agent 配置（Onboarding 旅程）

**角色：周楠，新加入团队的产品经理**
接替离职同事负责"B 端采购平台"项目。项目已运行 2 个月，有 4 个定制 Agent，但没有任何交接文档。

**场景展开：**

周楠进入项目 Agent 配置页，4 张 Agent 卡片显示创建者、创建时间、最后修改时间。她点开"采购产品 Agent"详情——版本历史（v1→v3，各版本有修改说明"增加合规审查要求"）清晰可读；"运行实例"标签显示过去 30 天该 Agent 被 6 个需求调用，已完成需求的输出质量有负责人标记。

**理解时刻**：20 分钟后，周楠知道这个 Agent"是什么"、"为什么是这样"、"用得怎么样"。她没有重新创建 Agent，而是在前人基础上做了一处微调——团队知识第一次实现了真正的传递。

**旅程揭示的需求：**
- F09 Agent 详情页（创建者/修改历史/版本说明）

- 运行实例标签：历史调用记录 + 输出质量标记
- "理解既有配置"是复用率 60% 目标的关键路径

## 创新与新颖模式

### 检测到的创新领域

**创新一：AI 能力的"工程化"定义范式**

现有 Agent 平台（Dify、Coze 等）通过自由文本提示词定义 Agent 行为，能力边界模糊、不可枚举。AI-DZHT 引入 **BMAD 命令库作为原子能力单元**——每个命令（BP/DR/CA 等）对应明确的研发阶段职责，Agent 能力由命令组合精确定义。这使得：
- Agent 能力可以被**机器查询**（能力反查 F12）
- 能力重叠可以被**自动检测**（F13）
- 新 Agent 的能力边界对团队成员**可读、可预期**

这是一种将 AI 能力从"文本描述"变为"结构化声明"的范式转变。

**创新二：三层上下文注入架构**

通用 Agent 平台的上下文注入是单层的（每次对话重新输入）。本模块提出三层分离设计：
- **层 1（Agent 层）**：角色定义与能力边界，跨所有使用场景不变
- **层 2（项目层）**：行业/技术栈/团队规范，同项目内复用
- **层 3（需求层）**：PRD 摘要/Story 列表，每次执行动态注入

三层各自维护、按需叠加，Agent 行为精准适配场景，同时避免了"为每个项目创建独立 Agent"的配置爆炸问题。

**创新三：研发角色预置 + Commander 动态编排**

区别于通用 Agent 平台的"用户自由组合"模式，本模块将 8 个研发角色 Agent 作为平台标准预置，Commander 在运行时动态决定调用顺序。用户不需要理解"什么时候该调哪个 Agent"——平台已将研发最佳实践编码进默认调度策略，用户只需配置"变体参数"，而非重新设计编排逻辑。

### 验证方向

| 创新点 | 验证假设 | 验证方式 |
|--------|----------|----------|
| 命令原子化 | 用命令定义的 Agent 比自由文本 Agent 更少产生"能力边界模糊"投诉 | A/B 对比：命令式 vs 自由文本 Agent 的用户反馈率 |
| 三层上下文 | 注入项目上下文后，Agent 输出"需补充背景"的频率显著降低 | 对比有/无项目上下文配置的需求执行质量评分 |
| 角色预置 | 用户首次配置完成率在预置角色引导下 ≥ 80% | MVP 上线后埋点统计 |

### 创新风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| 命令库扩展滞后 | 用户需要不在命令库里的能力时，无法通过标准路径配置 | P1 支持管理员扩展命令库（F02）；同时允许提示词自由补充 |
| 三层上下文冲突 | 项目层和需求层规范冲突时 Agent 行为不确定 | 明确优先级规则：需求层 > 项目层 > Agent 基础层 |
| 用户不理解分层 | 用户不清楚"在哪一层配置什么"，导致配置混乱 | UX 需清晰的层级可视化与配置引导 |

## SaaS B2B 平台专项需求

### 多租户模型

本模块采用**两级租户架构**：

```
组织/企业（Organization）
    └── 团队（Team）[1个组织可含多个团队]
            └── 项目（Project）
                    └── Agent 实例
```

**数据隔离规则：**
- Agent 模板默认可见范围为**团队级**——不同团队的 Agent 模板互相不可见
- 组织管理员可查看和管理组织内所有团队的 Agent 模板
- 跨团队共享需主动发布，支持三级范围控制（团队私有 / 组织内开放 / 申请使用）
- 项目上下文配置隔离在项目层，不跨项目共享

**设计决策：** 当前阶段（内部工具）不需要跨组织隔离，组织层为单例。待对外 SaaS 化时扩展为多组织支持。架构设计需预留 `tenant_id` 字段为后续 SaaS 化打基础。

### 平台内部集成依赖

| 依赖模块 | 集成方式 | 说明 |
|----------|----------|------|
| **大模型管理模块** | 内部 API 调用 | F05 AI 生成提示词草稿时调用；AI 服务可用性由该模块保障，本模块只消费接口 |

- 集成接口采用内部 API 契约，具体协议在架构设计（CA）阶段定义

## 项目范围与分期规划

### MVP 策略

**MVP 类型：体验驱动型 MVP（Experience MVP）**

不追求功能全覆盖，而是交付一条**完整的端到端价值体验**：项目负责人能在 10 分钟内完成"选能力 → AI 生成提示词 → 绑定项目上下文"的完整闭环，并立即感受到"Agent 知道我们是谁"的核心差异。

**MVP 支持的核心用户旅程：**
- ✅ 旅程 1：项目负责人首次配置（含 AI 草稿降级路径）
- ✅ 旅程 5：边缘保护（内置 Agent 不可删，运行中锁定）
- 部分覆盖旅程 2：研发成员查看（只读目录）

**MVP 不覆盖的旅程：** 旅程 3（冗余治理）、旅程 4（跨团队共享）、旅程 6（Onboarding）——在 Phase 2 和 Phase 3 补全。

### 交互设计原则（UX Principles）

> Agent 管理是**后台配置场景**，所有 UI/UX 决策以信息密度和操作效率为第一优先级，不追求视觉复杂度。

| ID | 原则 | 关联需求 |
|----|------|----------|
| NFR-UX01 | **配置优先，展示次之**：界面决策优先支持高效配置操作，减少不必要的动效与装饰性元素 | 全局 |
| NFR-UX02 | **操作意图明确**：停用、删除、锁定等高风险操作必须在执行前展示影响范围，系统主动告知，不依赖用户主动查询 | FR20 / FR21 / FR22 |
| NFR-UX03 | **降级不中断**：AI 辅助功能（提示词生成、单块优化）失败时，界面平滑切换至手动模式，展示清晰的操作引导而非技术错误信息 | FR42 |
