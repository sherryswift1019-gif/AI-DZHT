## 1. 现状基线

| 组件 | 状态 | 说明 |
|------|------|------|
| 用户数据模型 | ✅ 已有 | `users` 表含 id/username/display_name/email/role/status/avatar/created_at/updated_at |
| 用户 CRUD API | ✅ 已有 | 7 个端点（增删改查 + 启用/禁用），21 条 pytest 覆盖 |
| 密码字段 | ❌ 缺失 | 模型无 password_hash，无任何认证逻辑 |
| 登录/登出 | ❌ 缺失 | 无 JWT/Session/Cookie 机制，所有端点公开访问 |
| RBAC 中间件 | ❌ 缺失 | role 字段已定义但未做权限拦截 |
| 邀请注册 | ❌ 缺失 | 无邀请码/邀请链接机制 |
| 前端用户页面 | ❌ 缺失 | 无登录页、用户管理页、个人设置页 |
| E2E 测试基础设施 | ❌ 缺失 | 无 Playwright/Cypress 配置 |

**结论**：后端 CRUD 骨架可复用，但认证层、邀请机制、前端 UI、E2E 基础设施均需从零搭建。

## 2. 功能域拆解

### 2.1 用户 CRUD（增强现有）

| 子功能 | 当前 | 需要做 |
|--------|------|--------|
| 创建用户 | ✅ POST /api/v1/users | 增加 password 字段，密码哈希存储 |
| 查询用户列表 | ✅ GET /api/v1/users | 增加分页参数规范化、角色过滤 |
| 查询单用户 | ✅ GET /api/v1/users/{id} | 确保不返回 password_hash |
| 更新用户 | ✅ PATCH /api/v1/users/{id} | 支持修改密码（需验证旧密码） |
| 删除用户 | ✅ DELETE /api/v1/users/{id} | 增加权限校验（仅 admin） |
| 启用/禁用 | ✅ POST enable/disable | 增加权限校验（仅 admin） |

**风险点**：现有 4 条种子用户无密码，升级需兼容——首次登录强制设密码或提供迁移脚本。

### 2.2 账号密码登录

**核心问题清单**：

| # | 问题 | 候选方案 | 倾向 | 理由 |
|---|------|----------|------|------|
| 1 | Token 类型 | JWT / HttpOnly Cookie Session | **JWT** | 前后端分离架构，无需服务端 session 存储，与 SQLite 轻量定位一致 |
| 2 | 密码哈希 | bcrypt / argon2 / passlib | **passlib[bcrypt]** | Python 生态成熟，FastAPI 文档推荐，`pip install passlib[bcrypt]` 一行搞定 |
| 3 | Token 过期 | 短期 access + 长期 refresh / 单 token | **双 token** | access 30min + refresh 7d，兼顾安全与体验 |
| 4 | Token 存储（前端） | localStorage / memory + httpOnly cookie | **memory + httpOnly refresh cookie** | 防 XSS 窃取 refresh token |
| 5 | 登出机制 | 前端清 token / 后端黑名单 | **前端清除 + refresh token 轮换** | SQLite 无需维护黑名单表，轮换即可失效旧 token |

**需要新增的 API 端点**：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/auth/login` | POST | 用户名+密码登录，返回 access_token，set refresh_token cookie |
| `/api/v1/auth/refresh` | POST | 用 refresh cookie 换新 access_token |
| `/api/v1/auth/logout` | POST | 清除 refresh cookie |
| `/api/v1/auth/me` | GET | 返回当前登录用户信息 |
| `/api/v1/auth/change-password` | POST | 修改密码（需旧密码验证） |

### 2.3 邀请注册

**机制设计候选**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A: 邀请码 | admin 生成一次性邀请码，新用户用码注册 | 实现简单，可控 | 无法精准追踪谁邀请谁 |
| B: 邀请链接 | 系统生成含 token 的 URL，邮件/IM 分发 | 体验好，可追踪 | 需处理链接过期、发送渠道 |
| C: 管理员直接创建 | admin 在后台创建账号+临时密码，通知用户 | 最简单 | 临时密码传递不安全 |

**倾向方案 B（邀请链接）**，理由：
- 研发团队场景下，Slack/飞书分发链接成本极低
- token 机制可复用 JWT 签发逻辑
- 可设置 24h/48h 过期，安全可控

**邀请注册所需新增**：

| 组件 | 说明 |
|------|------|
| `invitations` 数据库表 | id, email, role, token, invited_by, expires_at, used_at, created_at |
| `POST /api/v1/invitations` | admin 创建邀请（指定 email + 角色） |
| `GET /api/v1/invitations` | admin 查看邀请列表（含状态） |
| `DELETE /api/v1/invitations/{id}` | admin 撤销邀请 |
| `POST /api/v1/auth/register` | 用邀请 token 注册（设用户名+密码） |
| 前端注册页 | `/register?token=xxx` 路由 |

### 2.4 前端页面清单

| 页面 | 路由 | 核心功能 |
|------|------|----------|
| 登录页 | `/login` | 用户名/密码登录，错误提示，"忘记密码"占位 |
| 注册页 | `/register` | 邀请 token 校验 → 填写用户名/密码/显示名 → 提交 |
| 用户管理页 | `/settings/users` | 用户列表 + 搜索/过滤 + 创建/编辑/删除/启用禁用（admin 可见） |
| 邀请管理页 | `/settings/invitations` | 生成邀请链接 + 邀请列表 + 状态追踪（admin 可见） |
| 个人设置页 | `/settings/profile` | 修改显示名/头像/密码 |

**前端基础设施需新增**：
- `useAuth` hook（登录态管理，token 刷新）
- `AuthGuard` 路由守卫组件
- `useUsers` hook（用户 CRUD 操作）
- `useInvitations` hook（邀请管理操作）
- Axios/fetch 拦截器自动附加 Authorization header

## 3. E2E 测试策略

### 3.1 框架选型

| 框架 | 优点 | 缺点 | 适配度 |
|------|------|------|--------|
| **Playwright** | 多浏览器、API 测试内置、auto-waiting、trace viewer | 较重 | ⭐⭐⭐⭐⭐ |
| Cypress | 社区大、调试体验好 | 单 tab 限制、跨域复杂 | ⭐⭐⭐ |
| Vitest Browser Mode | 与现有 Vitest 统一 | E2E 能力不成熟 | ⭐⭐ |

**倾向 Playwright**，理由：
1. 内置 API 测试能力可直接验证后端，无需额外工具
2. 项目规则要求"核心端点必须有自动化测试覆盖"，Playwright 可同时覆盖 API + UI
3. `page.request` 可在同一测试中混合 API 调用和 UI 操作，适合种子数据准备
4. trace viewer 对调试长流程测试极有价值

## 4. 安全考量

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 密码明文存储 | 数据泄露即全部沦陷 | passlib bcrypt 哈希，cost factor ≥ 12 |
| JWT 泄露 | 身份冒用 | 短期 access token（30min）+ httpOnly refresh cookie |
| 暴力破解 | 账号被撞库 | 登录失败计数，5 次锁定 15 分钟（可选 Phase 2） |
| 邀请 token 被猜测 | 未授权注册 | 使用 `secrets.token_urlsafe(32)`，48h 过期 |
| XSS 窃取 token | 会话劫持 | access token 仅存内存，refresh 走 httpOnly cookie |
| CSRF | 伪造请求 | SameSite=Lax cookie + 自定义 header 校验 |

## 5. 分阶段实施建议

| 阶段 | 范围 | 交付物 | 优先级 |
|------|------|--------|--------|
| **Phase 1: 认证基础** | 密码字段 + 哈希 + 登录/登出 API + auth 中间件 + 登录页 | 可登录的系统 | P0 |
| **Phase 2: 用户管理 UI** | 用户列表/创建/编辑/删除/启用禁用前端页面 + 权限校验 | admin 可管理用户 | P0 |
| **Phase 3: 邀请注册** | 邀请表 + 邀请 CRUD API + 注册页 + 邀请管理页 | 完整用户生命周期 | P1 |
| **Phase 4: E2E 测试** | Playwright 基础设施 + 全部测试用例 | 自动化回归保障 | P0（与 Phase 1-3 并行） |
| **Phase 5: 加固** | 登录限速、密码强度策略、操作审计日志 | 安全加固 | P2 |

## 7. 关键约束与项目规则对齐

| 项目规则 | 本模块对齐方式 |
|----------|----------------|
| 状态可追踪 | 邀请状态 pending → used / expired / revoked，用户状态 active / disabled |
| 禁用 ORM Session | 所有新表操作延续 SQLAlchemy Core `conn.execute()` 模式 |
| API 前缀 `/api/v1/` | 新增 `/api/v1/auth/*` 和 `/api/v1/invitations/*` |
| Pydantic BaseModel | 所有请求/响应模型用 Pydantic 定义 |
| 核心端点自动化测试 | Playwright E2E 覆盖全部用户管理场景，后端 pytest 补充 auth 单元测试 |
| 真实数据库测试 | E2E 测试连接真实 SQLite，禁止 mock 持久层 |
