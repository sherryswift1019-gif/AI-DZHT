# 头脑风暴：用户管理模块

## 1. 现状分析与缺口识别

| 维度 | 当前状态 | 缺口 |
|------|---------|------|
| 用户 CRUD | 后端 `/api/v1/users` 已实现基础增删改查 + 启用/禁用 | 缺少密码字段、缺少前端管理页面 |
| 登录认证 | **完全不存在** — 无密码存储、无 JWT/Session、无中间件 | 需从零构建认证链路 |
| 邀请注册 | 不存在 | 需设计邀请码/链接机制 + 注册流程 |
| 前端用户管理 | 无任何用户管理页面 | 需新建用户列表页、详情页、创建/编辑表单 |
| 权限控制 | 数据模型有 `role: admin|member|viewer` 但无运行时拦截 | 需路由守卫 + API 权限中间件 |
| E2E 测试 | 仅有后端 API 测试 (`test_users_api.py`, 211 行)，无浏览器级 E2E | 需引入 Playwright + 建立 E2E 基础设施 |
| MSW Mock | 已安装 `msw@2.13.3`，未配置 handler | 需配置 mock handler 支撑前端隔离测试 |

## 2. 功能维度拆解

### 2.1 用户 CRUD（增删改查）

**核心场景：**
- **C — 创建用户**：管理员手动添加用户，指定用户名、显示名、邮箱、角色、初始密码
- **R — 查看用户**：用户列表（分页/搜索/筛选）+ 用户详情卡片
- **U — 编辑用户**：修改显示名、邮箱、角色；管理员重置密码；用户自改密码
- **D — 删除用户**：软删除 vs 硬删除；关联数据处理（项目 owner/member 引用）

**延伸思考：**
- 用户名唯一性校验（后端已有 `get_user_by_username`）
- 批量操作：批量禁用/启用、批量角色变更
- 头像上传 vs 默认 Gravatar / 首字母头像
- 操作审计日志：谁在什么时间做了什么变更

### 2.2 账号密码登录

**核心场景：**
- 登录页面：用户名 + 密码表单
- 登录成功：签发 JWT / Session，前端存储 token，后续请求携带
- 登录失败：错误提示（用户不存在 / 密码错误 / 账号已禁用）
- 登出：清除 token，跳转登录页
- 会话保持：token 刷新 / 记住我

**延伸思考：**
- 密码策略：最小长度、复杂度要求
- 密码存储：bcrypt/argon2 哈希，绝不明文
- 登录频率限制：防暴力破解
- 多设备登录：允许 / 单端互踢
- 首次登录强制改密码

### 2.3 邀请注册

**核心场景：**
- 管理员生成邀请（指定角色 + 过期时间）
- 邀请形式：邀请码 / 邀请链接 / 邮件邀请
- 被邀请人通过邀请链接访问注册页
- 注册页：填写用户名、密码、显示名
- 注册完成自动登录并进入系统
- 邀请状态追踪：pending → accepted / expired

**延伸思考：**
- 邀请配额限制
- 邀请码一次性 vs 多次使用
- 无邮件服务时的降级方案：仅生成链接，管理员手动分享
- 公开注册 vs 仅限邀请注册（系统级配置开关）

### 2.4 权限与路由守卫

**核心场景：**
- 未登录用户访问任何页面 → 重定向到登录页
- Viewer 角色无法进行写操作
- Admin 专属功能：用户管理、邀请管理
- API 层权限校验中间件

## 3. 技术方案候选

### 3.1 认证方案对比

| 方案 | 适用场景 | 复杂度 | 推荐度 |
|------|---------|--------|--------|
| **JWT (无状态)** | 当前 SPA 架构天然匹配，前端 localStorage/cookie 存储 | 中 | **推荐** |
| Session + Cookie | 服务端有状态，需管理 session store | 中 | 适合但不如 JWT 契合 SPA |
| OAuth2 / OIDC | 需对接外部 IdP，架构过重 | 高 | 当前阶段不推荐 |

**推荐方案：JWT**
- Access Token (短期，15-30min) + Refresh Token (长期，7d)
- FastAPI 中间件 `Depends()` 注入当前用户
- 前端 React Query 拦截器自动刷新

### 3.2 密码存储方案

| 方案 | 安全性 | 性能 | 推荐度 |
|------|--------|------|--------|
| **bcrypt** | 高（自适应哈希） | 适中 | **推荐**（生态成熟） |
| argon2 | 更高（抗 GPU） | 可调 | 可选，依赖更多 |
| scrypt | 高 | 适中 | 可选 |

### 3.3 数据库 Schema 扩展

需要新增/修改的表：

```
users_table 扩展字段：
  + password_hash (String, NOT NULL)    -- bcrypt 哈希
  + last_login_at (Integer, nullable)   -- 最后登录时间戳
  + must_change_password (Boolean, default False)

invitations_table (新建)：
  - id (String, PK)
  - code (String, unique)              -- 邀请码
  - role (String, default "member")     -- 预设角色
  - created_by (String, FK → users.id)
  - status (String: pending|accepted|expired)
  - accepted_by (String, nullable, FK → users.id)
  - expires_at (Integer)               -- 过期时间戳
  - created_at (Integer)
```

## 5. 风险与约束

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 数据库 Schema 变更影响现有数据 | users 表新增 `password_hash` 列，现有 4 条种子数据需迁移 | 提供迁移脚本，为现有用户设置默认密码 |
| 认证引入后所有现有 API 行为变化 | 未携带 token 的请求将被拒绝，影响开发和测试流程 | 分阶段上线：先建认证模块，再逐步保护端点；测试用 fixture 自动获取 token |
| SQLite 无原生迁移工具 | 无 Alembic 等迁移管理，Schema 变更靠 `init_db()` | 在 `init_db()` 中增加 `ALTER TABLE` 兼容逻辑，或引入轻量迁移 |
| 前端无 E2E 基础设施 | 需从零搭建 Playwright 环境 | 尽早建立 E2E scaffold，CI 中先跑冒烟测试 |
| JWT Secret 管理 | 硬编码 secret 是安全隐患 | 使用环境变量，`.env` 不入库 |
| 密码存储安全 | bcrypt 计算耗时影响 API 响应 | 登录/注册端点允许更高超时，不影响其他 API |

## 6. 创新机会

1. **无密码登录（渐进增强）**：MVP 先做账号密码，后续可加 Magic Link / WebAuthn 作为可选登录方式
2. **Agent 操作身份绑定**：每个 Agent 执行动作绑定触发用户身份，实现操作溯源（谁触发了哪个 Agent 的哪次执行）
3. **团队空间**：用户管理 + 项目管理联动，实现项目级成员权限（Owner / Editor / Viewer）
4. **登录日志仪表板**：可视化登录次数、活跃时段、异常登录（与项目"实时可观测"目标一致）
5. **RBAC 策略引擎**：超越简单三角色，支持自定义权限策略（如"QA 只能操作测试相关 Agent"）

## 7. 模块依赖关系

```
邀请注册 ──依赖──→ 登录认证 ──依赖──→ 用户 CRUD
                      │
                      ▼
                 权限控制中间件
                      │
                 ┌────┴────┐
                 ▼         ▼
            前端路由守卫  API 权限校验
```

**推荐实施顺序：**
1. 用户 CRUD 补全（密码字段 + 前端页面）
2. 登录认证（JWT + 登录页 + 中间件）
3. 邀请注册（邀请表 + 注册流程）
4. 权限控制（路由守卫 + API 拦截）
5. E2E 测试（贯穿每个阶段同步编写）

## 8. 关键决策点（需后续 PRD 明确）

| 决策项 | 选项 | 建议 |
|--------|------|------|
| 登录方式 | 仅账号密码 / + Magic Link / + OAuth | MVP 仅账号密码 |
| 密码策略 | 无要求 / 8位+大小写 / 12位+特殊字符 | 8位+大小写数字 |
| 邀请形式 | 邀请码 / 邀请链接 / 邮件 | 邀请链接（无需邮件服务） |
| 注册开放性 | 仅邀请 / 公开注册+邀请 | 仅邀请（企业场景） |
| 删除策略 | 软删除 / 硬删除 | 软删除（保留审计轨迹） |
| Token 存储 | localStorage / httpOnly Cookie | httpOnly Cookie（更安全） |
| E2E 框架 | Playwright / Cypress | Playwright（更现代，跨浏览器） |
