# 用户管理模块 E2E 测试 — 头脑风暴报告

## 1. 现状分析

| 维度 | 当前状态 | 差距 |
|------|---------|------|
| 用户 CRUD | 已有 `GET/POST/PATCH/DELETE /api/v1/users`，含 disable/enable | 缺少密码字段、邮箱唯一约束 |
| 认证系统 | **不存在** — 所有 API 完全公开，无 JWT/Session/Cookie | 需从零搭建 |
| 权限控制 | users 表有 role 字段（admin/member/viewer），但路由无守卫 | 需新增中间件 |
| 邀请注册 | 不存在 | 需新增邀请表、邮件/链接流程 |
| 前端认证 | 无登录页、无 AuthContext、无路由守卫 | 需新增完整认证 UI 层 |
| 测试基础 | 仅 `tests/test_users_api.py` 覆盖基础 CRUD，无 E2E 框架 | 需引入 Playwright |

**关键发现**：数据库 users 表无 `password_hash` 字段，前端无任何认证状态管理。这意味着"登录"和"邀请注册"是 **全新功能**，不是对现有功能的测试。

## 2. 功能域拆解

### 2.1 用户 CRUD（增强现有）

| 功能点 | 说明 | E2E 测试关键场景 |
|--------|------|-----------------|
| 创建用户 | POST /api/v1/users，新增 password 字段 | 正常创建 → 数据库持久化验证 |
| 用户列表 | GET /api/v1/users，支持分页/筛选 | 列表渲染、空状态、角色筛选 |
| 用户详情 | GET /api/v1/users/{id} | 详情页数据完整性 |
| 编辑用户 | PATCH /api/v1/users/{id} | 修改角色/显示名后刷新验证 |
| 删除用户 | DELETE /api/v1/users/{id} | 删除确认弹窗 → 列表更新 |
| 禁用/启用 | POST /api/v1/users/{id}/disable\|enable | 状态切换后登录行为变化 |

**头脑风暴要点**：
- 当前预置了 4 个种子用户（含 admin: zhangshanshan），E2E 测试需处理种子数据的存在性
- username 已有唯一约束，需测试重复用户名的错误处理
- 删除 admin 自身需禁止或二次确认

### 2.2 账号密码登录（新建）

| 功能点 | 说明 | E2E 测试关键场景 |
|--------|------|-----------------|
| 登录页面 | 新增 /login 路由 | 页面渲染、表单校验 |
| 登录认证 | POST /api/v1/auth/login | 正确凭据 → Token 返回 |
| Token 管理 | JWT 或 Session | Token 存储、过期、刷新 |
| 登出 | 清除客户端凭据 | 登出后无法访问受保护页面 |
| 路由守卫 | 未认证 → 重定向到 /login | 直接访问 /projects 被拦截 |
| 禁用账号拒绝登录 | status=disabled 时拒绝 | 登录返回明确错误信息 |

**头脑风暴要点**：
- **认证方案选型**：JWT（无状态，适合 SPA）vs Session Cookie — 推荐 JWT，因项目是纯 SPA 架构，无服务端渲染
- **密码存储**：bcrypt/argon2 哈希，需在 users 表增加 `password_hash` 列
- **前端状态管理**：新增 Zustand authStore，存储 token + 当前用户信息
- **API 拦截器**：所有 API 请求自动附加 Authorization header，401 自动跳转登录页
- 需考虑 SSE 推流端点 `/api/v1/steps/stream` 的认证传递（EventSource 不支持自定义 header，需 query param token 或 cookie 降级）

### 2.3 邀请注册（新建）

| 功能点 | 说明 | E2E 测试关键场景 |
|--------|------|-----------------|
| 生成邀请 | Admin 创建邀请链接/码 | 邀请创建 → 链接可访问 |
| 邀请注册页 | /register?invite=xxx | 无效邀请码提示、过期提示 |
| 注册表单 | 用户名+密码+显示名 | 表单校验、用户名冲突 |
| 邀请消耗 | 注册后邀请码失效 | 同一码不可二次使用 |
| 邀请管理 | Admin 查看/撤销邀请 | 列表展示、撤销后注册失败 |

**头脑风暴要点**：
- **邀请模型**：新建 `invitations` 表（id, code, role, created_by, expires_at, used_by, used_at, status）
- **开放注册 vs 纯邀请制**：研发团队工具场景建议 **纯邀请制**，由 admin 控制准入
- 邀请链接格式：`{base_url}/register?invite={uuid}` — 简洁且可直接分享
- 邀请码有效期：默认 7 天，admin 可自定义
- 注册时预分配角色：邀请创建时指定 role，注册用户自动获得该角色

## 3. E2E 测试架构思考

### 3.1 技术选型

| 选项 | 推荐度 | 理由 |
|------|--------|------|
| **Playwright** | ★★★★★ | 官方推荐、支持多浏览器、API 测试能力强 |
| Cypress | ★★★☆☆ | 社区活跃但多浏览器支持弱 |
| Selenium | ★★☆☆☆ | 过重，不适合现代 SPA |

**推荐 Playwright**：与项目 Vite + React 技术栈天然契合，且支持 API + UI 混合测试。

### 3.2 测试环境策略

```
┌──────────────────────────────────────────────────┐
│  E2E 测试运行环境                                  │
│                                                    │
│  Playwright Test Runner                            │
│       │                                            │
│       ├── 启动 FastAPI 后端（测试 SQLite 实例）      │
│       ├── 启动 Vite Dev Server（前端）              │
│       └── 执行测试用例                              │
│                                                    │
│  每个测试文件前：清空并重新 seed 数据库              │
│  测试间隔离：每个 test 使用独立的浏览器 context      │
└──────────────────────────────────────────────────┘
```

**核心原则**（来自项目规则）：
- ✅ 使用真实 SQLite 数据库，禁止 mock 持久层
- ✅ 前端开发环境的 MSW mock 在 E2E 测试中应**关闭**，走真实 API
- ✅ 每个测试可追踪状态，测试结果可观测

### 3.3 测试用例矩阵（高层）

| 模块 | 正向场景 | 异常/边界场景 | 预计用例数 |
|------|---------|-------------|-----------|
| 用户 CRUD | 创建、查看列表、编辑、删除 | 重复用户名、删除自身、空列表 | 8-10 |
| 登录认证 | 正常登录、登出、Token 刷新 | 错误密码、禁用账号、过期 Token | 6-8 |
| 路由守卫 | 已登录访问、角色差异展示 | 未登录重定向、越权访问 | 4-5 |
| 邀请注册 | 创建邀请、注册成功、邀请消耗 | 过期邀请、无效码、重复注册 | 6-8 |
| **合计** | | | **24-31** |

## 4. 风险与约束

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| users 表需 schema 迁移（加 password_hash） | 现有种子数据需兼容 | 迁移脚本为现有用户设默认密码 |
| SSE 端点认证传递 | EventSource API 限制 | 支持 query param token 降级 |
| 前端全局改造（AuthContext + 路由守卫） | 影响所有现有页面 | 登录态封装为高阶组件，渐进式接入 |
| 测试数据隔离 | 并行测试可能互相干扰 | 每个 test 文件独立数据库文件 |
| data.db 保护 | 绝不能删除运行时数据库 | E2E 测试使用独立的 `test_e2e.db`，与 `data.db` 完全隔离 |

## 5. 发散思考：增值功能

以下功能不在核心需求范围内，但值得在 PRD 阶段考虑是否纳入：

| 想法 | 价值 | 复杂度 | 建议 |
|------|------|--------|------|
| 操作审计日志 | 谁在何时做了什么，与 Agent 事件日志对齐 | 中 | 推荐纳入 — 与项目"可观测"目标一致 |
| 密码强度策略 | 安全基线 | 低 | 推荐纳入 — 实现成本低 |
| 头像上传 | users 表已有 avatar 字段 | 中 | 延后 — 非核心 |
| OAuth/SSO 集成 | 企业场景需要 | 高 | 延后 — 先落地基础认证 |
| 多因素认证 (MFA) | 安全增强 | 高 | 延后 — 用户规模不大时 ROI 低 |
| 用户在线状态 | 团队协作感知 | 中 | 延后 — 与当前需求无关 |

## 6. 关键决策点（需下游确认）

1. **认证方案**：JWT access_token + refresh_token（推荐） vs HttpOnly Cookie Session
2. **密码哈希算法**：bcrypt（推荐，生态成熟） vs argon2（更安全但依赖更复杂）
3. **邀请制度**：纯邀请制（推荐） vs 开放注册 + 邮箱验证
4. **角色权限粒度**：现有 admin/member/viewer 三级是否满足需求，还是需要更细的权限矩阵
5. **种子用户迁移**：现有 4 个种子用户的默认密码策略（首次登录强制修改？）
6. **E2E 测试运行模式**：CI 集成（headless）还是先本地手动触发
