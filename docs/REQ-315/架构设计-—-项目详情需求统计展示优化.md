## 2. 现状分析

### 2.1 当前数据流

```
SQLite requirements_table
  │  status: queued | running | blocked | done
  │  pipeline: JSON[{status: PipelineStepStatus, ...}]
  ▼
FastAPI GET /api/v1/projects/{id}/requirements
  │  返回 Requirement[] (pipeline JSON 反序列化)
  ▼
useRequirementList(projectId)
  │  TanStack Query 管理缓存，running 状态下 5s 自动 refetch
  ▼
useMemo → statCounts  ← 前端实时计算 6 个互斥桶
  │  { queued, running, pendingApproval, pendingInput, blocked, done }
  ▼
StatChip × 6 + 总数标签  ← 当前 UI 展示
```

### 2.2 当前统计分类逻辑

```typescript
// ProjectDetailPage.tsx:130-151 — 已实现互斥分类
switch (r.status) {
  case 'done'    → done
  case 'blocked' → blocked
  case 'queued'  → queued
  case 'running' → {
    currentStep.status ∈ {pending_approval, pending_advisory_approval} → pendingApproval
    currentStep.status === 'pending_input'                             → pendingInput
    else                                                               → running
  }
}
// 恒等式：sum(6桶) === requirements.length ✓
```

### 2.3 当前 UI 布局（已实现双分组）

```
┌─ 介入组 (--attention-surface bg) ─┐  │  ┌─ 进度组 ──────────────┐
│ 待审批(warning) 阻塞(danger) 待输入(amber) │  │  │ 执行中(blue) 待执行(text-3) 已完成(success) │
└───────────────────────────────────┘  │  └────────────────────────┘  共 N
```

> **结论**：统计分类逻辑和双分组布局已在前序迭代中实现。本次优化聚焦于**展示风格和视觉层次**的微调。

## 3. 架构决策记录 (ADR)

### ADR-001: 统计计算保持纯前端 — 不引入后端聚合 API

| 项 | 内容 |
|----|------|
| **状态** | 已确认 |
| **上下文** | 当前 6 桶统计由前端 `useMemo` 从 `requirements[]` 实时计算，无独立聚合端点 |
| **决策** | 维持前端计算，不新增 `GET /api/v1/projects/{id}/requirement-stats` |
| **理由** | ① 单项目需求量级 <100，前端 O(n) 遍历性能可忽略；② 避免前后端重复维护分类逻辑；③ TanStack Query 已实现 5s 自动 refetch，实时性满足需求；④ 减少 API 表面积 |
| **风险** | 若单项目需求量级未来超过 1000+，可考虑后端聚合。当前阶段无此需求 |

### ADR-002: StatChip 保持内联函数 — 不抽取独立文件

| 项 | 内容 |
|----|------|
| **状态** | 已确认 |
| **上下文** | `StatChip` 当前是 `ProjectDetailPage.tsx` 底部的内联函数组件（8 行） |
| **决策** | 保持内联，不抽取为 `components/StatChip.tsx` |
| **理由** | ① 仅此一处使用；② 逻辑极简（纯展示，无状态）；③ 过早抽象增加文件碎片，违反项目 [AVOID] 原则 |
| **触发迁移条件** | 其他页面需要复用相同统计芯片时 |

### ADR-003: CSS Token 变更范围 — 仅在现有 Token 体系内调整

| 项 | 内容 |
|----|------|
| **状态** | 已确认 |
| **上下文** | `index.css` 已定义 `--attention-surface`、`--amber`、`--warning`、`--danger` 等 Token，覆盖 light/dark 双主题 |
| **决策** | 本次变更仅使用已有 Token，不新增 CSS 变量 |
| **理由** | 前序 UX 设计已将所需 Token 全部就位，视觉调整通过 Tailwind utility class 组合即可完成 |

## 7. 边界场景处理

| 场景 | 预期行为 | 验证方式 |
|------|----------|----------|
| 零需求项目 | 所有 StatChip 显示 `0`，总数显示 `共 0` | 新建空项目验证 |
| 全部完成 | 介入组全 `0`，已完成 = 总数 | 将所有需求状态置为 done |
| 高紧急度 | 介入组数字突出（暖色 + 背景色强调） | 构造多条 pending_approval 需求 |
| 单条需求 | 仅一个桶为 `1`，其余为 `0`，总数 `共 1` | 仅创建一条需求 |
| 统计求和 | `sum(6桶) === requirements.length` 恒成立 | 任意状态组合下检查 |

## 9. 验收标准

1. **互斥性**：6 个统计桶求和严格等于需求总数，任何状态组合下不重复不遗漏
2. **视觉分组**：介入组（待审批/阻塞/待输入）使用暖色背景容器与进度组视觉区隔
3. **双主题**：Light / Dark 主题下所有统计项颜色对比度满足 WCAG AA（≥ 4.5:1 文本）
4. **响应性**：统计栏在 ≥1280px 宽度下不换行，内容不截断
5. **实时性**：需求状态变更后统计数字在 ≤5s 内刷新（依赖现有 refetch 机制）
6. **零回归**：不影响需求列表、需求详情、流水线操作等既有功能
7. **零后端变更**：不新增 API 端点，不修改数据库 Schema
